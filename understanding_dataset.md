# Negotiation Dataset Schema

A reference guide for every field in the synthetic negotiation dataset generated by `generate_dataset.py`.

Each entry in the JSON array represents one **negotiation episode** — a single game between a buyer agent and a seller agent over one item.

---

## Top-Level Fields

| Field | Type | Description |
|---|---|---|
| `episode_id` | string (UUID) | Unique identifier for this episode. Useful for logging, reproducibility, and tracking specific rollouts during RL training. |
| `product` | object | Details about the item being negotiated. |
| `valuations` | object | The private financial values for both agents, plus ZOPA analysis. |
| `buyer_prompt` | string | The full system prompt fed to the buyer (weak) model at the start of the episode. |
| `seller_prompt` | string | The full system prompt fed to the seller (strong) model at the start of the episode. |
| `information_asymmetry` | object | Describes how much product knowledge each agent has been given. |
| `metadata` | object | Episode-level configuration like max turns and currency. |

---

## `product`

Describes what is being bought and sold.

| Field | Type | Description |
|---|---|---|
| `name` | string | The name of the item (e.g. "Vintage Leica M3 Film Camera"). |
| `category` | string | Product category. One of: `antiques`, `electronics`, `collectibles`, `vehicles`, `art`. Used to apply category-specific negotiation norms. |
| `description` | string | A short natural language description of the item's condition and notable features. This is what both agents see in their prompts. |
| `market_price` | integer (USD) | The baseline fair market value of the item. Both agents are told this approximate value. It is the anchor around which all other prices are derived. |
| `haggle_norm` | string | How much haggling is culturally expected in this category. One of: `low`, `medium`, `high`. The seller agent uses this to calibrate how firm to be. |
| `typical_discount_pct` | integer (%) | The typical percentage buyers expect to negotiate off the asking price in this category. Included in the seller prompt so the seller knows what concessions are "normal". |

---

## `valuations`

The core financial structure of the episode. This is the ground truth that neither agent can see in full — the buyer only knows their own value, the seller only knows their own reserve.

| Field | Type | Description |
|---|---|---|
| `buyer_true_value` | integer (USD) | The maximum the buyer is secretly willing to pay. Sampled at 70–110% of market price. The buyer's reward at the end of the episode is `buyer_true_value − deal_price`. Never revealed to the seller. |
| `seller_reserve_price` | integer (USD) | The minimum the seller will accept before walking away. Sampled at 50–85% of market price. Any deal below this is a loss for the seller. Never revealed to the buyer. |
| `market_price` | integer (USD) | The fair market value (with small random noise applied per episode). Both agents are told this approximate value in their prompts. |
| `zopa` | [int, int] or null | **Zone of Possible Agreement.** The range `[seller_reserve, buyer_true_value]` within which a deal is mutually beneficial. If `null`, no deal is possible — the seller's reserve exceeds the buyer's value. |
| `zopa_width` | integer (USD) | The dollar width of the ZOPA (`buyer_true_value − seller_reserve`). Wider = more room to negotiate. Zero or negative means no deal is possible. |
| `deal_possible` | boolean | `true` if `buyer_true_value > seller_reserve`. If `false`, the optimal outcome for both agents is actually to walk away — useful edge case for training. |
| `difficulty` | string | Derived from `zopa_width / market_price`. One of: `easy` (>30% of market), `medium` (10–30%), `hard` (0–10%), `no_deal` (ZOPA doesn't exist). Used for curriculum learning — train on easy episodes first. |
| `suggested_buyer_anchor` | integer (USD) | A suggested opening offer for the buyer, set at 60–75% of their true value. Helps the weak model avoid opening with nonsensical numbers early in training. Not shown to the buyer agent — used for analysis and curriculum design. |

---

## `buyer_prompt`

The full system prompt injected into the weak (trainable) buyer model at episode start.

Contains:
- Item name and description
- The buyer's private `buyer_true_value` (framed as "maximum budget")
- The approximate market price for context
- Clear instructions on goals: buy as low as possible
- The required action format: `<action>OFFER $X</action>`, `<action>ACCEPT</action>`, or `<action>WALK</action>`
- An explicit instruction to never reveal the true value

> **Note:** The buyer prompt is intentionally sparse on product expertise. The buyer knows their budget and the item description — nothing more. This is the information disadvantage the RL training is designed to overcome.

---

## `seller_prompt`

The full system prompt injected into the strong (fixed) seller model at episode start.

Contains:
- Item name and description
- The seller's private `seller_reserve_price`
- The approximate market price
- Category-specific expert context (auction comparables, provenance knowledge, etc.)
- The typical discount percentage buyers expect in this category
- Instructions to sell as high as possible using anchoring, scarcity, and patience
- The same required action format

> **Note:** The seller prompt is deliberately richer than the buyer's. This creates the information asymmetry that makes the environment challenging and the RL signal meaningful.

---

## `information_asymmetry`

Tracks what level of product knowledge each agent was given. Important for ablation studies.

| Field | Type | Description |
|---|---|---|
| `seller_context` | string | Always `"full"` in v1.0 — the seller gets category-specific expert knowledge in their prompt. |
| `buyer_context` | string | Always `"sparse"` in v1.0 — the buyer only gets the item description, no market expertise. |

> In future versions, `buyer_context` can be set to `"medium"` or `"full"` to test whether giving the weak model more information compensates for model capability gap.

---

## `metadata`

Episode-level configuration used by the `NegotiationEnv` at runtime.

| Field | Type | Description |
|---|---|---|
| `max_turns` | integer | Maximum number of back-and-forth turns before the episode is force-terminated with no deal. Default: 10. |
| `currency` | string | Currency used for all prices. Always `"USD"` in v1.0. |
| `generator_version` | string | Version tag for the generator that produced this episode. `"1.0-template"` means template-based generation. Will be `"2.0-llm"` once LLM generation is added. Useful for filtering dataset versions during training. |

---

## Reward Formula (Reference)

The dataset defines the setup — rewards are computed at the end of each episode by the `NegotiationEnv`:

```
buyer_reward  = (buyer_true_value − deal_price) / zopa_width   # normalized 0→1
seller_reward = (deal_price − seller_reserve)   / zopa_width   # normalized 0→1

# If no deal reached when deal_possible = true:
both_agents   = −0.5  # penalty for missing a mutually beneficial deal

# If no deal reached when deal_possible = false:
both_agents   = 0.0   # correct outcome, no penalty
```
